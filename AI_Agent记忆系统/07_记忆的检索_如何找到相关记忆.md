# AI Agent 记忆系统 - 第 7 课：记忆的检索——如何找到相关记忆

## 内容讲解

### 检索的本质问题

第 4 课我们学了 RAG 的基本流程，第 6 课学了"写什么"。现在的问题是：

> 写入的记忆可能有几千甚至几万条，面对用户的一个提问，如何**快速、准确**地找到最相关的那几条？

这比看起来难得多。因为用户的提问方式千变万化，同一个意图可以有无数种表达。

### 检索的完整流水线

```
用户输入："上次那个性能问题后来怎么解决的？"
    │
    ▼
┌──────────────────────────┐
│ 阶段一：查询预处理         │
│                          │
│  1. 查询理解：提取意图     │
│     → 想找：性能问题的解决方案
│                          │
│  2. 查询改写/扩展         │
│     → "性能问题 解决方案"  │
│     → "性能优化"          │
│     → "响应慢 修复"       │
│                          │
│  3. 元数据提取            │
│     → 时间线索："上次"     │
│     → 类别线索："技术问题" │
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│ 阶段二：多路召回           │
│                          │
│  路线 A：向量搜索          │
│    查询向量 → Top 20      │
│                          │
│  路线 B：关键词搜索        │
│    BM25("性能") → Top 20  │
│                          │
│  路线 C：元数据过滤        │
│    时间：最近 30 天         │
│    类别：技术问题           │
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│ 阶段三：合并与重排序       │
│                          │
│  1. 合并去重              │
│  2. 重排序模型打分         │
│  3. MMR 多样性调整         │
│  4. 时间衰减加权           │
│  → 最终 Top 5             │
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│ 阶段四：结果注入           │
│                          │
│  将 Top 5 记忆片段插入     │
│  上下文窗口中              │
└──────────────────────────┘
```

我们逐阶段展开。

### 阶段一：查询预处理

用户的原始提问往往不是最佳的搜索查询。预处理的目的是**把用户意图转换为更有效的搜索信号**。

#### 查询改写（Query Rewriting）

第 4 课我们简单提过，现在深入看：

```
原始查询："上次那个性能问题后来怎么解决的？"

改写策略：

1. 去口语化：
   "性能问题的解决方案"

2. 同义扩展：
   "性能优化方案" / "响应时间改善" / "速度提升方法"

3. 假设性回答（HyDE）：
   假设记忆中的答案可能是这样的：
   "性能问题通过增加缓存和优化数据库查询解决了"
   → 用这个假设答案的向量去搜索，往往比用问题本身更准
```

**HyDE（Hypothetical Document Embedding）** 是一个很巧妙的技术：不是用"问题"去找"答案"，而是先假设一个答案，用假设答案的向量去匹配。因为记忆中存储的通常是陈述句（答案），而不是疑问句（问题），所以假设答案的向量往往更接近真实记忆。

#### 元数据提取

从查询中提取结构化的过滤条件：

```
"上次讨论 Python 项目时"
  → 时间：最近的一次
  → 主题：Python 项目

"张三说的那个方案"
  → 人物：张三
  → 类型：方案/提案
```

### 阶段二：多路召回

不依赖单一检索方式，而是**多种策略并行**，然后合并结果。

#### 策略对比

| 策略 | 原理 | 擅长 | 不擅长 |
|------|------|------|--------|
| **向量搜索** | 语义相似度 | "数据库"找到"MySQL" | 精确标识符匹配 |
| **关键词搜索（BM25）** | 词频匹配 | 精确匹配 "PR #4521" | 同义词、换种说法 |
| **元数据过滤** | 结构化字段筛选 | 时间、类别、人物精确筛选 | 无法理解语义 |
| **知识图谱遍历** | 实体关系链 | "张三的部门的项目"这类多跳查询 | 需要预先构建图谱 |

#### 混合搜索的权重

OpenClaw 使用的混合搜索就是向量 + BM25 的加权组合：

```
最终得分 = α × 向量相似度得分 + (1-α) × BM25 得分

α 的典型值：0.5 ~ 0.7（偏向语义搜索）
```

这个 α 值可以根据查询类型动态调整：
- 查询包含特定标识符（ID、代码符号）→ 降低 α，加大关键词权重
- 查询是模糊描述 → 提高 α，加大语义权重

### 阶段三：合并与重排序

多路召回后，可能有 40-60 个候选结果。需要精选出最终的 Top-K。

#### 重排序模型（Reranker）

第一阶段的搜索（向量/BM25）是**粗排**——快但不够精确。重排序模型做**精排**：

```
粗排（快，处理全量数据）：
  向量搜索 / BM25
  → 40 个候选

精排（慢但准，只处理候选）：
  Reranker 模型（如 Cohere Rerank、bge-reranker）
  输入：(查询, 候选文档) 对
  输出：精确的相关性分数
  → 重新排序，选 Top 5
```

为什么需要两阶段？
- 向量搜索快但**对细微差异不敏感**（它比的是方向，不是精确语义）
- Reranker 精确但**处理不了海量数据**（需要逐对评估）
- 两者组合：用快方法缩小范围，用精方法精确排序

#### MMR（最大边际相关性）

第 10 课讲 OpenClaw 时会详细学到。核心思想：

> 在选结果时，不仅考虑**相关性**，还考虑**多样性**。

如果 Top 5 里有 3 条说的是同一件事，不如换掉其中 2 条，用虽然相关性略低但提供不同信息的结果。

#### 时间衰减

OpenClaw 的做法：

```
衰减得分 = 原始得分 × e^(-λ × 天数)

λ = 衰减系数（越大衰减越快）
```

效果：一周前的记忆得分打约 8 折，一个月前打约 3 折（取决于 λ 的设定）。

### 检索失败的模式

即使流水线设计得再好，检索仍然可能失败：

#### 模式一：沉默失败（Silent Failure）

检索返回了结果，但**没有一条是真正相关的**，Agent 却基于这些错误结果生成了回答。

> 这是最危险的情况——Agent 看起来很自信，但答案是基于无关记忆编造的。

应对：设置**相关性阈值**，得分低于阈值的结果不使用，Agent 应当承认"我没有找到相关记忆"。

#### 模式二：遗漏（Miss）

相关记忆存在于数据库中，但**没有被召回**。

常见原因：
- 查询和记忆的表达方式差异太大
- 记忆的分块方式导致关键信息被切断
- 元数据标注不准确

应对：查询改写 + 多路召回 + 定期审查分块质量。

#### 模式三：过载（Overload）

召回了太多结果，全部塞进上下文后反而干扰了 Agent 的判断。

应对：严格控制 Top-K 的数量（通常 3-5 条），质量优先于数量。

---

## 小结

1. 记忆检索是一个**四阶段流水线**：查询预处理 → 多路召回 → 合并重排序 → 结果注入。
2. **查询预处理**（改写、HyDE、元数据提取）决定了搜索的起点质量。
3. **多路召回**（向量 + 关键词 + 元数据）比单一策略更鲁棒。
4. **粗排 + 精排**两阶段设计平衡了速度和精度；MMR 和时间衰减进一步优化结果。
5. 要警惕三种检索失败模式：**沉默失败**（最危险）、**遗漏**、**过载**。

## 我的答案与纠偏记录

### 问题 1 我的回答
因为答案通常是陈述句,答案找答案更具备语义上的相似性。问题向量偏向疑问句,可能不会出现与答案相关的语义;假设答案向量,更大可能性跟真实答案有会相同的语义。

**纠偏**：完全正确，"答案找答案更具备语义相似性"表述精炼。具体例子：记忆中存着"通过增加 Redis 缓存层，API 响应时间从 2s 降到了 200ms"，用问题"性能问题怎么解决的？"的向量去搜语义距离较远，用 HyDE 假设答案"性能问题通过增加缓存解决了"的向量去搜语义距离很近。

### 问题 2 我的回答
因为精排模型不擅长处理大量的数据,需要依靠粗排手法先将相关性高的语料采集一遍。

**纠偏**：正确。补充原因：精排模型（Reranker）需要把每个 (查询, 文档) 对一起输入做交叉注意力计算，复杂度 O(n)，10 万条逐条精排延迟和成本不可接受。这种"先粗后精"在很多领域都有：搜索引擎（倒排索引→排序模型）、推荐系统（召回层→排序层）、人类认知（先扫一眼书架→再仔细看几本）。

### 问题 3 我的回答
在系统提示词层面,指导智能体,不要盲目采纳Top的语料,因为Top的语料可能相关性也是非常低的。同时,可以代码编码的方式,对低于阈值的相关性的语料直接过滤,这样就能避免大模型不严格执行提示词的问题。

**纠偏**：非常好，体现了**纵深防御**思维——代码层硬过滤（相关性阈值）+ 提示词层软引导，两层保障。特别是"代码编码方式过滤可以避免大模型不严格执行提示词"——**不要把安全关键的判断完全交给 LLM 的提示词遵循能力**。
