## 内容讲解

### 长期记忆的核心挑战

短期记忆的问题我们已经很清楚了：容量有限、会话后消失。长期记忆要解决的是：

> **如何把大量信息持久化存储，并在需要时快速找到相关的那一小部分？**

这个问题拆成两步：
1. **存**——用什么格式、什么结构存储？
2. **取**——给定当前问题，怎么找到相关记忆？

目前最主流的方案就是 **向量数据库 + RAG**。

### 第一步：文本 → 向量（Embedding）

#### 什么是 Embedding？

Embedding（嵌入）是把文本转换成一组**高维浮点数**的过程。

```
"我喜欢喝咖啡" → [0.12, -0.45, 0.78, 0.33, ..., -0.21]  （如 1536 维）
"我爱喝拿铁"   → [0.11, -0.43, 0.80, 0.31, ..., -0.19]  （非常接近！）
"明天天气怎样" → [0.89, 0.12, -0.56, 0.02, ..., 0.67]  （差距很大）
```

关键性质：**语义越相近的文本，向量越接近。**

#### 类比理解

想象一个二维坐标系（实际是 1536 维，但原理一样）：

```
        美食 ↑
             │    ·"咖啡"
             │  ·"拿铁"
             │
             │         ·"烹饪"
             │
    ─────────┼──────────────→ 技术
             │
             │              ·"Python"
             │           ·"编程"
             │
```

同一领域的概念在向量空间中**自然聚集**。这就是语义搜索的基础。

#### Embedding 模型

常用的 Embedding 模型：

| 模型 | 提供商 | 维度 | 特点 |
|------|--------|------|------|
| text-embedding-3-small | OpenAI | 1536 | 性价比高，通用 |
| text-embedding-3-large | OpenAI | 3072 | 更精确，成本更高 |
| voyage-3 | Voyage AI | 1024 | 代码和技术文档表现好 |
| BGE-M3 | BAAI | 1024 | 开源，多语言 |

### 第二步：向量数据库（Vector Database）

有了向量之后，需要一个专门的数据库来**高效存储和检索**它们。

#### 为什么不能用普通数据库？

```sql
-- 普通 SQL 数据库擅长的查询：
SELECT * FROM users WHERE age > 25 AND city = 'Beijing'

-- 但它无法做到的查询：
SELECT * FROM memories WHERE meaning ≈ '用户上次提到的那个编程问题'
```

向量数据库专门优化了**相似度搜索**——给定一个查询向量，快速找到最相似的 K 个向量。

#### 常见向量数据库

| 名称 | 类型 | 特点 |
|------|------|------|
| **Pinecone** | 云服务 | 全托管，开箱即用 |
| **Weaviate** | 开源/云 | 支持混合搜索 |
| **Chroma** | 开源 | 轻量级，适合原型开发 |
| **Qdrant** | 开源/云 | 高性能，支持过滤 |
| **FAISS** | 库（Meta） | 不是数据库，是向量索引库 |
| **sqlite-vec** | SQLite 扩展 | 轻量，OpenClaw 可选用 |

#### 相似度度量

两个向量有多"接近"？常见度量方式：

- **余弦相似度（Cosine Similarity）**：衡量方向的一致性，最常用
  - 1.0 = 完全相同方向，0 = 正交，-1.0 = 完全相反
- **欧氏距离（L2 Distance）**：直线距离
- **点积（Dot Product）**：结合了方向和大小

大多数场景用**余弦相似度**就够了。

### 第三步：RAG——把检索和生成串起来

**RAG（Retrieval-Augmented Generation）**，检索增强生成，是当前 Agent 长期记忆的核心模式。

#### 完整流程

```
用户提问："上次我们讨论的数据库迁移方案，最后选的是哪个？"
    │
    ▼
[1. 查询理解]
    把用户问题转换为向量
    │
    ▼
[2. 检索（Retrieval）]
    在向量数据库中搜索最相似的记忆片段
    返回 Top-K 个结果（如 K=5）
    │
    │  结果1: "2月15日讨论了三个迁移方案：A渐进式、B停机迁移、C双写..."
    │  结果2: "2月18日决定采用方案A（渐进式），因为不能停机..."
    │  结果3: "方案A的执行步骤：第一步建立同步管道..."
    │  ...
    ▼
[3. 注入上下文]
    把检索到的记忆片段插入到上下文窗口中（第3课讲的 ② 区域）
    │
    ▼
[4. 生成（Generation）]
    LLM 基于（用户问题 + 检索到的记忆）生成回答
    │
    ▼
输出："上次我们讨论了三个方案，最终在2月18日决定采用方案A（渐进式迁移），
      主要原因是系统不能停机。执行步骤是..."
```

#### 分块（Chunking）：一个容易被忽视的关键步骤

在建立向量索引之前，需要把长文档切成小块。为什么？

- Embedding 模型有输入长度限制（如 8K tokens）
- 太大的文本块，语义会被"平均化"，搜索不精准
- 太小的文本块，缺少上下文，检索到了也看不懂

```
一篇 5000 字的会议记录
    │
    ▼  分块
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
│ 块1:      │ │ 块2:      │ │ 块3:      │ │ 块4:      │
│ 400 tokens│ │ 400 tokens│ │ 400 tokens│ │ 400 tokens│
│ (含重叠)  │ │ (含重叠)  │ │ (含重叠)  │ │ (含重叠)  │
└──────────┘ └──────────┘ └──────────┘ └──────────┘
```

分块的常见参数：
- **块大小（Chunk Size）**：每块约 200-1000 tokens，OpenClaw 用的是 ~400
- **重叠（Overlap）**：相邻块之间重叠 50-100 tokens，避免信息在块边界被切断

### RAG 的常见问题

RAG 不是银弹。实际使用中会遇到这些问题：

#### 问题一：检索不准（Relevance Gap）

用户问的和存储的，语义上有微妙偏差：
> 存储："团队决定使用 PostgreSQL"
> 查询："我们选了什么数据库来着？"
> → "数据库" 和 "PostgreSQL" 的向量距离可能不够近

**改进**：混合搜索（向量 + 关键词），正是 OpenClaw 的做法。

#### 问题二：检索到了但不相关（False Positive）

语义相似 ≠ 实际相关：
> 存储："Python 的内存管理使用引用计数"
> 查询："Agent 的记忆管理"
> → "内存/记忆"和"管理"语义接近，但完全不是一回事

**改进**：元数据过滤（按时间、类别、来源筛选），重排序模型（Reranker）。

#### 问题三：上下文碎片化（Context Fragmentation）

分块导致完整信息被切断：
> 原文："方案的优点是[...块 1 结束...][块 2 开始...]成本低且实施周期短"
> 只检索到块 2，缺少"优点是"的前文

**改进**：增加块重叠，使用父文档检索（检索到子块后，返回整个父文档段落）。

## 小结

1. **Embedding** 把文本转为向量，语义相近的文本在向量空间中彼此接近。
2. **向量数据库**专为高效相似度搜索设计，是长期记忆的基础设施。
3. **RAG** 的流程是：查询理解 → 向量检索 → 注入上下文 → LLM 生成。
4. **分块**是一个容易被忽视但影响巨大的环节——块大小和重叠策略直接影响检索质量。
5. RAG 的三大常见问题：**检索不准、误召回、上下文碎片化**，各有对应的改进手段。

## 我的答案与纠偏记录

**掌握判定：完全掌握**

### 问题 1 我的回答
因为关键词无法索引到其他相关性的词汇,例如数据库关键词,匹配不一定能检索到MySQL,而在向量搜索时,这两个词的语义相关性非常强,向量层面余弦相似度很高,所以能被检索到。

**纠偏**：完全正确，例子恰当。

### 问题 2 我的回答
如果分块太小,则容易检索到内容但是内容不连贯出现断层;分块太大,则容易把语义平均化,没有明显的特征。OpenClaw选用~400 tokens,我认为这是他的应用场景决定的,他的场景大量都是对话,一次聊天的内容一般在~400tokens左右。

**纠偏**：完全正确，且从应用场景角度的分析有独立思考。补充：400 tokens 还平衡了检索精度（块越小语义越聚焦）和上下文完整性（400 tokens 大致能包含一个完整段落或观点）。

### 问题 3 我的回答
采用混合搜索方式,即关键词+向量相似检索,避免用词不一样导致无法检索成功。另外再结合元数据过滤,例如发送时间,发送对象,商品对象等等,更好的精准过滤出结果。

**纠偏**：完全正确，元数据例子非常具体。补充进阶手段：**查询改写（Query Rewriting）**——Agent 可以先把查询改写为多个变体（如"投诉"→"反馈""不满意"），同时提取用户ID作为元数据过滤条件，提升召回率。
