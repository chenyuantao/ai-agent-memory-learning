## 内容讲解

### 从 v1 到 v2：更成熟的记忆架构

上一课我们了解了 OpenClaw 的基本记忆架构。OpenClaw 的记忆系统还有一套更完善的 v2 设计（Workspace Memory v2），引入了更结构化的机制。

v2 的核心理念不变——**Markdown 是权威的、可审查的真相源**。但新增了一个派生存储层：

```
~/.openclaw/workspace/
├── memory/
│   └── YYYY-MM-DD.md       ← 每日日志（权威源）
├── bank/
│   ├── entities/            ← 实体记录（人、项目、概念...）
│   ├── world.md             ← 世界事实
│   ├── experiences.md       ← 经验教训
│   └── opinions.md          ← 带置信度的观点
└── .memory/
    └── index.sqlite         ← 派生索引（随时可重建）
```

关键设计原则：**index.sqlite 永远可以从 Markdown 文件重新构建**。数据库是为了加速查询，而不是真相源。

### 操作循环：Retain → Recall → Reflect

这是 v2 的核心运作模型，灵感来自 MemGPT/Letta 的控制循环和 Hindsight 的记忆基底：

#### 1. Retain（保留）——记住什么

每日日志中会生成一个 `## Retain` 段落，用特定前缀标记不同类型的信息：

```markdown
## Retain

- W: 用户的公司本月从 AWS 迁移到了 GCP  @用户公司
- B: 使用 streaming API 处理大文件比一次性读取更稳定  @文件处理
- O [0.6]: 用户似乎更喜欢函数式编程风格  @用户偏好
```

| 前缀        | 含义                             | 示例                         |
| --------- | ------------------------------ | -------------------------- |
| `W`       | World（世界事实）                    | "用户的公司使用 Python 3.11"      |
| `B`       | Best practice / Experience（经验） | "处理 CSV 时 pandas 比手动解析更可靠" |
| `O [置信度]` | Opinion（观点）                    | "[0.6] 用户偏好简洁的代码风格"        |

注意 `@entity` 标签——这是**实体关联**，让信息和具体的人、项目、概念挂钩。

#### 2. Recall（召回）——怎么找

v2 提供多种召回方式，比纯语义搜索丰富得多：

| 召回方式 | 说明 | 使用场景 |
|----------|------|----------|
| **词法搜索** | FTS5 全文检索 | 精确关键词匹配 |
| **实体查询** | 按 `@entity` 标签查找 | "关于@张三的所有记忆" |
| **时间查询** | 按日期范围筛选 | "上周的记忆" |
| **观点查询** | 按类型 + 置信度筛选 | "关于用户偏好的高置信度观点" |

搜索返回的结果包含**源文件引用**，Agent 可以追溯到原始 Markdown 查看完整上下文。

#### 3. Reflect（反思）——记忆的自我更新

这是最有意思的部分。OpenClaw 有一个**定时任务**，负责：

- **更新实体摘要**：根据新的每日日志，更新 `bank/entities/` 中的实体页面
- **调整观点置信度**：当新证据出现时，观点的置信度会变化

观点演化的规则很有意思：

> **"小幅调整；大幅跳变需要强有力的反驳。"**

举个例子：

```
初始：O [0.6] 用户偏好函数式编程风格
  ↓ 用户在三次对话中都使用了函数式写法
更新：O [0.75] 用户偏好函数式编程风格
  ↓ 用户明确说"我其实更喜欢面向对象"
更新：O [0.3] 用户偏好函数式编程风格
```

这就像人类的信念更新：如果只有轻微的不一致，你不会完全改变看法；但如果出现了强有力的反驳证据，你会大幅调整。

### CLI 操作实战

OpenClaw 通过 `memory` 命令管理记忆系统：

```bash
# 查看记忆系统状态
openclaw memory status

# 深度检查（探测向量 + 嵌入可用性）
openclaw memory status --deep

# 深度检查并在需要时重建索引
openclaw memory status --deep --index

# 建立/更新索引
openclaw memory index
openclaw memory index --verbose    # 显示详细的批处理日志

# 语义搜索
openclaw memory search "用户的编程语言偏好"
openclaw memory search --agent myagent "上周的会议记录"

# 禁用记忆功能
# 在配置中设置: plugins.slots.memory = "none"
```

### OpenClaw vs MemGPT：设计对比

| 维度       | OpenClaw         | MemGPT/Letta |
| -------- | ---------------- | ------------ |
| **存储哲学** | 文件即真相，数据库是派生     | 数据库是核心存储     |
| **可审查性** | 打开文件夹即可查看所有记忆    | 需要通过 API 查询  |
| **记忆分类** | W/B/O 前缀 + @实体标签 | 核心记忆 + 档案记忆  |
| **观点演化** | 显式置信度 + 定时反思     | 通过上下文隐式调整    |
| **版本管理** | 天然 Git 友好        | 需要额外方案       |
| **适用场景** | 个人助手、可审查场景       | 通用 Agent 框架  |

OpenClaw 的设计可以说是**借鉴了 MemGPT 的控制循环思想**，但在存储层面走了一条完全不同的路——选择了透明性和人类可读性。

### 实践建议（来自 OpenClaw 官方文档）

OpenClaw 团队对记忆系统的落地给出了务实的建议：

1. **从 `bank/` 实体页面 + SQLite FTS 开始**——这已经能覆盖大部分场景
2. **暂时跳过高级嵌入方案**——只在数据量很大时才需要
3. **深度集成到已有工作区**——利用 OpenClaw 已有的路径感知和会话模型

## 小结

1. **Retain-Recall-Reflect 循环**是 OpenClaw v2 记忆系统的核心运作模型，分别对应"记住什么""怎么找""如何更新"。
2. **结构化记忆银行（Bank）** 将记忆分为世界事实（W）、经验（B）、观点（O），并通过 @实体标签建立关联网络。
3. **观点带有置信度**，并通过定时反思机制随新证据演化——小幅渐变，大幅跳变需要强反驳。
4. **派生索引永远可重建**，保持了"Markdown 为真相源"的核心不变。
5. 与 MemGPT 相比，OpenClaw **牺牲了一些查询性能，换取了透明性、可审查性和 Git 友好性**。

## 我的答案与纠偏记录

**掌握判定：完全掌握**

### 问题 1 我的回答
Retain对应记忆的存储,解决应该记忆什么内容的问题;Recall对应记忆的回忆,解决应该回忆哪些内容的问题;Reflect对应记忆的更新,解决哪些记忆需要更新的问题。

**纠偏**：完全正确，三个阶段的核心问题概括精准。

### 问题 2 我的回答
因为很多观点不是绝对长期的,需要根据用户的持续使用来逐步加强,如果没有置信度,就会因为用户某次选用了另一种语言或者操作系统,就认为用户的偏好是那个语言或操作系统,偏好一直在变就等于没有记忆。

**纠偏**：完全正确。"偏好一直在变就等于没有记忆"一针见血——没有置信度的系统会在不同偏好之间反复横跳，最终记忆变成噪声。

### 问题 3 我的回答
数据库为真相源的话,就会不够透明,不方便编辑和审查;但是优势是查询性能会更强。我认为一些B端领域,需要很强的知识检索能力的场景,可以采用这种方式。

**纠偏**：完全正确，场景判断准确。补充：两者也可以共存——OpenClaw v2 的架构正是这样做的：Markdown 是真相源，SQLite 是派生索引。日常检索用数据库加速，审查和版本管理走 Markdown，唯一代价是需要维护两者之间的同步一致性。
