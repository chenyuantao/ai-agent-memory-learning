## 内容讲解

### 回顾：工作记忆 ≠ 短期记忆

第 2 课我们建立了一个关键区分：

| | 短期记忆 | 工作记忆 |
|---|---|---|
| 功能 | 被动存储 | **存储 + 主动操作** |
| 类比 | 便利贴 | 白板（可以在上面写写画画、推理） |
| 生命周期 | 当前会话 | 当前**任务** |

今天我们深入看工作记忆在 Agent 中到底是怎么实现的。

### 实现方式一：思维链（Chain of Thought, CoT）

这是最基础的工作记忆形式——**让模型把中间推理步骤说出来**。

#### 无 CoT vs 有 CoT

```
用户：一个书架有 3 层，每层放 12 本书，卖掉了总数的 1/4，还剩多少本？

无 CoT：
  答：还剩 27 本。

有 CoT：
  总共 3 × 12 = 36 本     ← 中间结果 1
  卖掉 36 × 1/4 = 9 本    ← 中间结果 2
  剩余 36 - 9 = 27 本      ← 最终结果
  答：还剩 27 本。
```

CoT 的本质：**把工作记忆的内容外显化**，写进短期记忆（上下文窗口）中。

模型不是在"脑中"同时保持 36 和 9 然后计算——它是把这些中间结果**写出来**，然后在后续 token 生成时可以"看到"之前写的内容。这就是为什么 CoT 能提升推理能力——**它用短期记忆的空间模拟了工作记忆**。

#### CoT 的局限

但这有一个你在第 3 课第 3 题中已经意识到的问题：

> 中间推理步骤写进上下文，会导致 tokens 快速膨胀。

对于简单数学题还好，但对于复杂 Agent 任务——比如分析 10 篇论文并交叉对比——CoT 的中间推理可能占据大量上下文空间。

### 实现方式二：Scratchpad（草稿板）

Scratchpad 是对 CoT 的升级——**给 Agent 一个专门的、结构化的工作空间**。

#### 和 CoT 的区别

```
CoT（混在对话流中）：
  用户：帮我对比方案 A 和方案 B
  Agent：让我先分析方案 A... 方案 A 的优点是... 缺点是...
         然后方案 B... 方案 B 的优点是... 缺点是...
         对比来看... [继续在对话流中推理]

Scratchpad（独立的工作空间）：
  ┌─ Scratchpad ────────────────────────┐
  │ 任务：对比方案 A 和 B                 │
  │ 进度：3/4                            │
  │                                      │
  │ 方案 A：                             │
  │   优点：成本低、部署快                 │
  │   缺点：扩展性差                      │
  │                                      │
  │ 方案 B：                             │
  │   优点：扩展性强                      │
  │   缺点：成本高、学习曲线陡             │
  │                                      │
  │ 待完成：撰写对比总结                   │
  └──────────────────────────────────────┘
```

关键区别：
- **结构化**：不是自由文本，而是有明确的格式（进度、已完成、待完成）
- **可更新**：不是追加式的，可以修改之前写的内容
- **任务导向**：明确绑定到一个具体任务

#### 实际 Agent 框架中的 Scratchpad

很多 Agent 框架通过系统提示词来实现 Scratchpad：

```
[System] 你有一个 <scratchpad>，用于规划和跟踪多步任务。
每次行动前，先更新 scratchpad。格式如下：

<scratchpad>
目标：[当前最终目标]
计划：
  1. [步骤] - [状态：完成/进行中/待做]
  2. ...
当前发现：
  - [关键发现]
下一步：[即将执行的操作]
</scratchpad>
```

### 实现方式三：外部工作记忆（External Working Memory）

更激进的方案——**把工作记忆完全移到上下文窗口之外**。

```
                     上下文窗口
                    ┌──────────┐
                    │ 系统提示  │
                    │ 当前对话  │
  外部工作记忆       │          │
  ┌───────────┐    │   ┌──┐   │
  │ 任务状态   │◄──►│   │读│   │
  │ 中间结果   │    │   │写│   │
  │ 待办清单   │    │   └──┘   │
  │ 推理草稿   │    │          │
  └───────────┘    └──────────┘
  （文件/数据库）
```

Agent 通过**工具调用**来读写外部工作记忆：
- `scratchpad_write("发现方案A成本更低")`
- `scratchpad_read()` → 返回当前草稿板内容
- `task_update(step=3, status="completed")`

优点：
- **不占用上下文窗口**
- 可以存储更多中间状态
- 任务中断后可以恢复

缺点：
- 额外的工具调用开销
- Agent 需要"学会"何时读写草稿板

### 工作记忆的容量瓶颈

人类的工作记忆有著名的 **7±2** 限制——同一时间只能处理约 5-9 个信息块。

Agent 的工作记忆也有类似的限制，虽然原因不同：

| 人类 | Agent |
|------|-------|
| 神经生物学限制 | 上下文窗口中的**注意力稀释** |
| ~7 个信息块 | 取决于信息块大小和窗口容量 |
| 超载时犯错 | 超载时**遗忘中间步骤、跳过分支、输出不一致** |

实验表明，当 Agent 需要同时追踪的状态超过一定数量时，性能会显著下降。这就是为什么好的 Agent 框架会鼓励：

1. **分解任务**：把大任务拆成小子任务，每个子任务的工作记忆负载更小
2. **及时归档**：完成一个子任务后，把结果写入长期记忆，释放工作记忆
3. **显式状态管理**：用 Scratchpad 而不是隐式的 CoT

### 三种记忆的协作全景（更新版）

```
用户："帮我写一篇关于远程办公的调研报告"

[工作记忆 - Scratchpad]
  目标：完成远程办公调研报告
  计划：
    1. 搜集数据 ✅
    2. 整理论点 → 进行中
    3. 撰写初稿
    4. 润色定稿
  当前发现：
    - 73% 的员工希望混合办公
    - 远程办公降低 20% 办公成本

       ↕ 读写                    ↕ 检索/存储

[短期记忆 - 上下文窗口]          [长期记忆 - 向量数据库]
  当前正在处理的论点               之前调研过的相关资料
  用户的格式要求                   用户的写作偏好
  最近的工具调用结果               类似报告的模板
```

## 小结

1. **思维链（CoT）** 是最基础的工作记忆实现，本质是用短期记忆的空间模拟工作记忆，优点是简单，缺点是占用上下文。
2. **Scratchpad** 提供结构化、可更新的工作空间，比 CoT 更适合复杂的多步任务。
3. **外部工作记忆**把状态移到上下文之外，不占用窗口空间，但引入工具调用开销。
4. Agent 的工作记忆也有容量限制——注意力稀释导致追踪过多状态时性能下降。
5. 好的实践是：**分解任务 + 及时归档 + 显式状态管理**。

**纠偏**：方向正确。具体做法：①归档已完成项——子任务完成后，把结果摘要写入长期记忆，从 Scratchpad 中移除详细过程 ②只保留活跃状态——Scratchpad 只保留"当前步骤"和"待完成项" ③分级详细度——当前步骤详细，其他步骤只留标题和结论。关键原则：**Scratchpad 应该像白板——写满了就擦掉已完成的，擦掉的内容去长期记忆。** 这是工作记忆和长期记忆之间的归档循环。
